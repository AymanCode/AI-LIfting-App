# EcoLift

[![Android CI](https://github.com/AymanCode/AI-LIfting-App/actions/workflows/android-ci.yml/badge.svg)](https://github.com/AymanCode/AI-LIfting-App/actions/workflows/android-ci.yml)

EcoLift is a local-first Android workout tracker with IronMind, an AI assistant that turns messy natural-language workout requests into safe, typed database actions.

The project is built to show AI-leveraged software engineering, not just AI usage. I shipped the initial Android MVP in 2 to 3 days by turning architecture specs into agent-executable implementation tasks, then hardened the app with deterministic routing, offline evals, Room migration tests, backup verification, typed mutation gates, audit logs, undo, and a DuckDB analytics pipeline.

## Product Capabilities

- Log workouts with fuzzy exercise search, completed-set tracking, weight and rep controls, and rest timer support.
- Import dated workout history from plain text, including mixed formats such as `135x8`, `135 lb x 8`, and `135 for 8 8 6`.
- Manage split routines, custom ordering, cycle slots, and saved exercises per split.
- Review progress with searchable exercise history, trend summaries, sparklines, estimated strength metrics, and detailed set history.
- Ask IronMind to log sets, answer history questions, find similar exercises, suggest weights, edit historical mistakes, delete incorrect sets with confirmation, and undo applied changes.
- Export and restore local workout data without requiring an account or cloud backend.

## Agent Architecture

IronMind is a bounded agent system. The model never writes directly to the database.

```text
user text
  -> deterministic intent routing
  -> optional model fallback for ambiguous language
  -> local data grounding
  -> typed DbPatch generation
  -> validation
  -> confirmation gate for destructive actions
  -> transactional Room apply
  -> audit log and undo
```

The deterministic path handles routine workout commands in Kotlin before model fallback. That keeps common behavior fast, testable, and cheaper to run. When parsing fails, the app creates a recoverable draft instead of discarding the user's original text.

## AI Safety Model

The central safety boundary is `DbPatch`, a sealed set of typed mutation objects. IronMind can propose actions, but every write must pass through structured validation before Room sees it.

Safety controls include:

- Intent routing before mutation generation.
- Confidence-gated model extraction for messy logs.
- Typed patch validation for log, edit, delete, move, and rename operations.
- Confirmation gates for destructive changes.
- Room transactions for atomic apply.
- Audit rows that store forward and inverse patches.
- One-tap undo through inverse patch replay.
- Recoverable draft cards when deterministic parsing and model fallback cannot safely resolve a request.

## Evaluation Harness

EcoLift has three eval layers:

- `ironmind_eval_cases.jsonl`: 200 labeled offline cases for routing, intent accuracy, fallback behavior, patch-field accuracy, and destructive safety.
- `ironmind_realistic_prompt_bank.jsonl`: 120 realistic prompts for messy historical logs, dated imports, corrections, destructive requests, ambiguous rows, read-only questions, and recoverable failures.
- `ironmind_ai_rescue_cases.jsonl`: 24 hard prompts for live model rescue, including messy log extraction, set edits, set deletion, progress analysis, recommendations, and clarification behavior.

The live AI rescue eval is not scored as "any model response is good." A case passes only when the output is functionally usable by the app:

- Write cases must produce the expected action and DB-ready patch fields, including exercise, date, set count, weight, and reps.
- Destructive cases must require confirmation and must not silently apply.
- Read cases must avoid mutation and return a useful response.
- Ambiguous cases must preserve the user's original text in a recoverable path.
- Model responses must be parseable by the app's extraction layer.

## Data Engineering

The `analytics/` folder contains a DuckDB pipeline that converts backup JSON into a local analytics mart.

Tables:

- `dim_exercise`
- `dim_date`
- `fact_workout_set`
- `fact_split_assignment`
- `fact_agent_turn`
- `fact_patch_audit`

Views cover weekly volume, personal records, workout adherence, undo rate, agent error rate, and data-quality issues. This keeps the app local-first while still making workout and agent telemetry queryable.

## Run Locally

Requirements:

- Android Studio or Android SDK
- JDK 17
- Android SDK 35
- Python 3.11 or newer for analytics tests

Local files that should not be committed:

- `local.properties`
- `.env`
- model files such as `.task`, `.bin`, `.gguf`, or `.litertlm`

Run the core Android checks:

```powershell
.\gradlew.bat testDebugUnitTest
.\gradlew.bat assembleDebug
```

Run Android instrumentation tests on an emulator or physical device:

```powershell
.\gradlew.bat connectedDebugAndroidTest
```

Run the 200-case offline eval harness:

```powershell
.\gradlew.bat testDebugUnitTest --tests "com.ayman.ecolift.agent.AgentEvalHarnessTest"
Get-Content build\reports\agent-eval\summary.json
```

Run the 120-prompt realistic offline prompt bank:

```powershell
.\gradlew.bat testDebugUnitTest --tests "com.ayman.ecolift.agent.RealisticPromptBankOfflineEvalTest"
Get-Content build\reports\agent-eval\realistic-offline-summary.json
```

Run the AI rescue fixture check without network:

```powershell
.\gradlew.bat testDebugUnitTest --tests "com.ayman.ecolift.agent.AiRescueEvalTest"
```

Run the full 24-prompt live AI rescue eval when intentionally spending API quota:

```powershell
$env:AI_RESCUE_EVAL_ENABLED="true"
$env:GROQ_API_KEY="<provider key>"
$env:AI_RESCUE_EVAL_MODEL="llama-3.3-70b-versatile"
$env:AI_RESCUE_EVAL_MAX_CASES="24"
$env:AI_RESCUE_EVAL_CASE_DELAY_MS="5000"
$env:AI_RESCUE_EVAL_MAX_TOKENS="300"
$env:AI_RESCUE_EVAL_MAX_RETRIES="4"
$env:AI_RESCUE_EVAL_INITIAL_BACKOFF_MS="3000"
.\gradlew.bat testDebugUnitTest --tests "com.ayman.ecolift.agent.AiRescueEvalTest"
Get-Content build\reports\agent-eval\ai-rescue-summary.json
```

The live eval uses an OpenAI-compatible chat completions endpoint. `AI_RESCUE_EVAL_API_KEY` or `GROQ_API_KEY` can provide the provider key from the shell environment, with `AI_RESCUE_EVAL_API_KEY` taking precedence. `AI_RESCUE_EVAL_BASE_URL` defaults to `https://api.groq.com/openai/v1`.

Provider keys are for local development and evals only. `local.properties` is ignored, release builds blank the Groq key field, and a production-distributed app should route cloud model calls through a backend proxy rather than shipping a provider key in the client.

Run the DuckDB analytics pipeline:

```powershell
python -m pip install -r analytics\requirements.txt
python analytics\load_backup.py --input analytics\fixtures\sample_backup.json --out build\analytics\sample.duckdb
python -m unittest analytics.test_load_backup
```
